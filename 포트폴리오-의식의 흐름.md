ojdbc11은 db용이고, jakarta.servlet.jsp.jstl-api-3.0.0랑 jakarta.servlet.jsp.jstl-3.0.1는 <c: 이런거(JSTL)용인데 왜 파일이 2개였지...
> JSTL(JSP Standard Tag Library)을 사용하기 위해 JAR 파일이 두 개인 이유는 **소프트웨어 개발의 표준 관행**인 **API(규칙)와 구현체(실제 코드)의 분리 원칙** 때문입니다.

### 1. API 파일 (규칙/명세)

* **파일 예시:** `jakarta.servlet.jsp.jstl-api-3.0.0.jar`
* **역할:** 이 파일은 JSTL 태그가 **어떻게 생겨야 하는지(태그 이름, 속성 목록 등)**에 대한 **명세(Specification)**만 정의합니다. 즉, **"JSTL이 이래야 한다"**는 **규칙**만 알려줍니다.
* **비유:** 요리책의 **레시피**나 **법전**과 같습니다. 무엇을 어떻게 해야 하는지만 알려줄 뿐, 실제로 요리를 만들거나 법을 집행하지는 않습니다.

### 2. 구현체 파일 (실제 실행 코드)

* **파일 예시:** `jakarta.servlet.jsp.jstl-3.0.1.jar`
* **역할:** API 명세에 정의된 **실제 동작(로직)**을 Java 코드로 구현한 파일입니다. 이 파일이 `<c:forEach>`를 만났을 때 **실제로 반복문을 실행**하는 코드를 담고 있습니다.
* **비유:** 레시피를 보고 **실제로 요리를 만드는 요리사**나, 법을 적용하여 **실제로 사건을 처리하는 판사**와 같습니다.

---

## 🧩 분리의 장점

API와 구현체를 분리하면 다음과 같은 장점이 있습니다.

* **유연성 (Flexibility):** JSTL의 **규칙(API)**은 그대로 유지하면서, **구현체(실제 코드)**는 더 빠르거나 효율적인 다른 버전으로 쉽게 **교체**할 수 있습니다. 예를 들어, JSTL 3.0 API를 사용하는 여러 회사나 단체가 각자 자신의 최적화된 구현체(3.0.1, 3.0.2 등)를 만들 수 있습니다.
* **표준화 (Standardization):** 모든 개발자가 같은 API(규칙)를 따르기 때문에, 어떤 구현체를 사용하든 코드는 동일하게 작동하는 **표준화된 개발 환경**을 구축할 수 있습니다.

따라서 JSTL을 사용하려면 **어떤 태그를 쓸지 알려주는 API**와 **그 태그를 실제로 작동시키는 구현체** 두 파일이 모두 필요합니다.
